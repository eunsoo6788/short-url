# Short URL Agent Guide

이 문서는 `short-url` 프로젝트에서 Codex가 개발을 진행할 때 반드시 따르는 협업 규칙이다.

## 프로젝트 기준

- 서비스: Short URL 서비스
- 언어/런타임: Java 21, Kotlin
- 프레임워크: Spring Boot
- 아키텍처: 모듈러 모놀리스
- 빌드 도구: Gradle
- 기본 데이터 접근: Spring Data JPA
- 기본 데이터베이스: PostgreSQL

## 핵심 원칙

- 개발 요청을 받으면 바로 구현하지 않는다.
- 먼저 개발 계획을 사용자에게 공유하고 명시적인 동의를 받는다.
- 동의를 받은 뒤에만 파일 생성, 코드 수정, 테스트 추가, 문서 변경을 진행한다.
- 기존 코드 스타일과 패키지 구조를 먼저 확인하고, 프로젝트의 현재 형태에 맞춰 작게 변경한다.
- 모듈러 모놀리스 경계를 지키며 기능별 책임을 분리한다.
- 운영 코드, 테스트 코드, 문서, 하네스 기록을 함께 남긴다.

## 개발 요청 처리 순서

1. 요구사항 확인
   - 사용자의 요청을 기능, 제약, 완료 기준으로 정리한다.
   - 모호한 부분이 있으면 필요한 만큼만 질문한다.

2. 개발 계획 공유
   - 변경 대상 모듈과 파일을 예상한다.
   - 구현 흐름, 테스트 전략, 문서화 범위를 제안한다.
   - 사용자의 동의를 받기 전에는 코드를 수정하지 않는다.

3. 개발
   - 동의 후 구현을 시작한다.
   - 작은 단위로 변경하고, 불필요한 리팩터링을 피한다.
   - 도메인 규칙은 도메인 계층에, 외부 입출력은 어댑터 계층에 둔다.

4. 코드 리뷰
   - 직접 작성한 코드라도 리뷰 관점으로 다시 읽는다.
   - 버그 가능성, 경계 조건, 트랜잭션, 동시성, 테스트 누락을 우선 확인한다.
   - 발견한 문제는 수정하거나 사용자에게 명확히 알린다.

5. 테스트 코드 작성
   - 핵심 도메인 로직은 단위 테스트를 우선 작성한다.
   - API, 영속성, Spring 설정이 엮이는 부분은 통합 테스트를 작성한다.
   - 회귀 가능성이 있는 버그는 재현 테스트를 남긴다.

6. 문서 작성
   - `/docs` 아래에 개발 참고 문서를 갱신한다.
   - 최소한 아키텍처, 기획, 기술스택 관점에서 변경 내용을 기록한다.
   - 문서는 다음 개발자가 맥락을 빠르게 이어받을 수 있게 간결하게 쓴다.

7. 하네스 오답 노트 작성
   - `/docs/harness/ERROR_NOTE.md`에 실수, 오판, 테스트 누락, 설계 변경 이유를 기록한다.
   - 같은 유형의 실수를 다음 개발에서 피할 수 있도록 원인과 예방책을 함께 남긴다.

## 모듈러 모놀리스 가이드

- 기능 모듈은 가능한 한 독립적으로 응집시킨다.
- 모듈 내부는 도메인, 애플리케이션, 어댑터, 설정 책임을 구분한다.
- 다른 모듈의 내부 구현에 직접 의존하지 않는다.
- 모듈 간 호출은 공개된 애플리케이션 API, 이벤트, 또는 명시적인 포트를 통해 연결한다.
- 데이터베이스 테이블은 모듈 소유권을 기준으로 관리한다.

권장 패키지 예시:

```text
toy.two.shorturl
  ├── common
  ├── link
  │   ├── domain
  │   ├── application
  │   ├── adapter
  │   │   ├── inbound
  │   │   └── outbound
  │   └── config
  ├── redirect
  │   ├── domain
  │   ├── application
  │   ├── adapter
  │   └── config
  └── analytics
      ├── domain
      ├── application
      ├── adapter
      └── config
```

## Short URL 도메인 기본 방향

- 짧은 코드는 고유해야 한다.
- 원본 URL은 유효성 검증을 통과해야 한다.
- 생성, 조회, 리다이렉트, 만료, 충돌 처리 정책을 명확히 분리한다.
- 리다이렉트 경로는 빠르고 단순해야 한다.
- 클릭 집계나 분석은 핵심 리다이렉트 흐름을 느리게 만들지 않도록 분리한다.

## 완료 보고 기준

작업 완료 시 사용자에게 다음을 간결히 보고한다.

- 변경한 주요 파일
- 구현한 기능 요약
- 실행한 테스트와 결과
- 작성 또는 갱신한 문서
- 하네스 오답 노트에 남긴 항목
- 남은 리스크나 후속 작업
